## Project Vision

**RShopRefills V2** is a production-grade multi-vendor marketplace and payment platform.

Core goals:

- Multi-vendor marketplace (vendors list products, customers buy/refill)
- Manual verification + billing workflows
- Wallet system for vendors and customers
- Admin dashboard (SaaS-style, dashboard-heavy)
- Pusher real-time integrations + email notifications
- Vendor settlement / payout workflows

---

## Team & Workflow Rules

- Team project, multiple developers
- Branch workflow: feature branches into develop into main
- Never commit directly to main
- Never force push
- Never hardcode local DB credentials or expose production keys
- Each developer uses their own local database

### Role split

- **Roddy Tech (frontend engineer):** owns UI / design only. Blade, Tailwind, Flux/Volt markup, Alpine state, copy. Reads backend files to learn contracts, never modifies them.
- **Backend team:** owns controllers, models, migrations, routes, middleware, validation rules, mail config, `.env`.

### Frontend workflow

Before any UI task:

1. Check what the backend team has already implemented (Volt PHP blocks, controllers, routes, models, validation rules, events).
2. Then do exactly one of two things:
   - **Build frontend only** when no backend exists yet. Design the markup with sensible `wire:model` / form field names so backend can wire it up later.
   - **Wire frontend to existing backend** when backend exists. Bind to the existing properties, methods, validation messages, routes, and events. Match what is there exactly.

Never modify PHP logic, validation rules, route definitions, middleware, models, or `.env`. If something on the backend looks broken or missing, flag it as FYI for the backend team and stop.

## Design Conventions

Team-wide rules that apply to every Blade / CSS / view file:

- **Brand color:** solid blue (`blue-600` primary, `blue-700` hover, `blue-950` dark panels). No purple, no violet.
- **No gradients.** Use solid colors only. Avoid `bg-gradient-to-*`, `from-*`, `via-*`, `to-*` and inline gradients.
- **No em or en dashes** in code, copy, comments, or commit messages. Use commas, periods, parentheses, or rewrite the sentence.
- **Copyright year:** hardcoded `2026`. Do not use `date('Y')`.
- **Action button arrows:** size `w-5 h-5` (20px), not the default `w-4 h-4`.
- **Storefront is always light mode.** `@fluxAppearance` is intentionally omitted from the storefront layout so Flux never injects a dark class.
- **Page surround color:** `#dbeafe` (Tailwind `blue-100`) — used as the body + content area background on both admin and customer dashboards. Replaces the older `#f1f6fb` (admin) and `#f5f7fb` (customer) tones.
- **Badges:** all badges across the site use `rounded-[5px]` (not `rounded-full`) with a **solid saturated background + white text** (e.g. `bg-blue-600 text-white`, `bg-emerald-500 text-white`, `bg-amber-500 text-white`, `bg-red-500 text-white`). Status pills follow the same rule (no -50/-100 tint + dark text combos).
- **Card style:** `rounded-2xl` (or `rounded-[20px]` on admin), `bg-white`, `shadow-sm shadow-zinc-900/5 ring-1 ring-zinc-100`. Icon tiles inside cards use `bg-blue-100` (light) or `bg-XXX-200` (admin KPI cards for stronger contrast against the blue-100 page bg).
- **Sidebar angle:** content area has `rounded-tl-[20px]` against a white outer to create a visible "tab" where the sidebar meets the content. Both admin and customer dashboards use this.

## Architecture Conventions

### Laravel 12 Structure

- No `Kernel.php`. Middleware registered in `bootstrap/app.php`.
- No `app/Console/Kernel.php`. Console routes in `routes/console.php`.
- Service providers in `bootstrap/providers.php`.

### Frontend / UI Pattern

- **Flux components** for the dashboard / admin area: `<flux:sidebar>`, `<flux:navlist>`, `<flux:button>`, `<flux:menu>`.
- **Volt single-file components** for interactive pages (PHP logic + Blade in one `.blade.php`).
- **Blade anonymous components** for static layout pieces and shared partials.
- **Alpine.js** for client-side state on the storefront (modals, dropdowns, transitions).
- Sidebar layout as the primary dashboard shell.

### Routing

- Named routes everywhere. Use `route('name')`, not hardcoded URLs.
- Volt routes via `Volt::route('path', 'component-name')`.
- Auth routes isolated in `routes/auth.php`.
- Web routes in `routes/web.php`.

### Auth system separated by guard (done, uncommitted)

Backend shipped a multi-auth architecture in `4c42c50` + `f0e88cd`: a separate `Admin` model + `admin` guard + `AdminAuthService` + `AdminAuth` middleware + dedicated `routes/admin.php`. Our frontend was still routing `/admin/*` content pages under the `web` guard, which would let any logged-in customer hit admin URLs. This session aligned the frontend with the backend's separation.

**Two completely separate auth flows, isolated sessions:**

| | Customer (`web` guard) | Admin (`admin` guard) |
|---|---|---|
| Model | `User` | `Admin` (separate table, role enum) |
| Login URL | `/login` (Volt) | `/admin/login` (Blade form) |
| Auth handler | Volt component, `Auth::attempt()` | `AdminAuthService` (rate limit, session regen, deactivation check, audit log) |
| Middleware | `auth + verified` | `admin` (re-checks `is_active` per request) |
| Routes file | `routes/web.php` + `routes/auth.php` | `routes/admin.php` |
| Dashboard | `/dashboard` (welcome stub) | `/admin/dashboard` (KPIs) |
| Content pages | none (customer area is the storefront) | `/admin/{products,orders,customers,transactions,wallets}` |
| Layout | `<x-layouts.app.header>` | `<x-layouts.admin>` |
| Logout | `route('logout')` | `route('admin.logout')` |

**Concrete changes this session:**

- `routes/web.php`: removed the misrouted `/admin/*` block (orders, customers, transactions, wallets were under `auth + verified`, wrong guard — would have leaked the admin layout to any verified customer).
- `routes/admin.php`: added 5 view routes (`admin.products`, `admin.orders`, `admin.customers`, `admin.transactions`, `admin.wallets`) inside the existing `admin` middleware group, alongside the shipped `admin.dashboard`.

Active-state highlighting via `request()->routeIs('admin.orders*')` etc. works end-to-end. The 6 product-category subitems (gift-cards, eSIMs, mobile-topups, bill-payments, flights, stays) and 4 bottom items (reports, marketing, support-tickets, settings) stay as `href="#"` until those views ship.
- Deleted 3 unused Laravel starter-kit auth layouts (`components/layouts/auth/{split,card,simple}.blade.php`). Only `centered.blade.php` is actively used (by forgot/reset/verify/confirm pages).

**Customer auth flows (login/register/forgot/reset/verify/confirm) were already aligned** with backend conventions — stock Laravel patterns matching the `User` model. ## Dev Environment Notes

- Run dev server: `composer run dev` (PHP server + queue + Vite together).
- Or individually: `php artisan serve` + `npm run dev`.
- DB: MySQL. Set your own credentials in `.env`, not committed.
- Default `.env` DB: `DB_DATABASE=app`, `DB_USERNAME=root`, `DB_PASSWORD=`.
- If frontend changes do not appear: `npm run build` or `npm run dev`.
- **Be consistent with `localhost` vs `127.0.0.1`** when developing locally. Switching between them mid-session breaks OAuth state validation and any other session-scoped flow.
- **Windows PHP CA bundle** (one-time setup): download `cacert.pem` from <https://curl.se/ca/cacert.pem>, save to a stable location, then add `curl.cainfo` and `openssl.cafile` to `php.ini` pointing at that file. The Google sign-in controller has a local-only workaround that disables SSL verify, but installing the CA bundle is the proper fix.
